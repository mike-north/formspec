## Methodology

The source inventory started with:

```bash
rg -n "from ['\"]typescript['\"]|import \* as ts from ['\"]typescript['\"]" packages/*/src
```

That raw grep is intentionally only a starting point: in this tree it also matches the
explanatory comment in `packages/eslint-plugin/src/utils/type-utils.ts`. The final count
and table coverage were verified from actual `ImportDeclaration` nodes with this AST
check:

```bash
node <<'NODE'
const fs = require("node:fs");
const path = require("node:path");
const ts = require("typescript");

function walk(dir, out = []) {
  for (const ent of fs.readdirSync(dir, { withFileTypes: true })) {
    const p = path.join(dir, ent.name);
    if (ent.isDirectory()) walk(p, out);
    else if (/\.tsx?$/.test(ent.name)) out.push(p);
  }
  return out;
}

const packages = fs
  .readdirSync("packages")
  .filter((name) => fs.existsSync(path.join("packages", name, "src")));
const sites = [];
for (const pkg of packages) {
  for (const file of walk(path.join("packages", pkg, "src"))) {
    const sf = ts.createSourceFile(file, fs.readFileSync(file, "utf8"), ts.ScriptTarget.Latest, true);
    for (const stmt of sf.statements) {
      if (
        ts.isImportDeclaration(stmt) &&
        ts.isStringLiteral(stmt.moduleSpecifier) &&
        stmt.moduleSpecifier.text === "typescript"
      ) {
        sites.push(file);
      }
    }
  }
}
console.log(sites.sort().join("\n"));
console.log(`count=${sites.length}`);
NODE
```

`ts_apis_used` was also checked with the TypeScript parser so both value-space property
access and type-space qualified names such as `ts.Program` were counted. Enum-like
families are aggregated as `SyntaxKind (N)`, `TypeFlags (N)`, etc., where `N` is the
number of distinct enum members used in that file. Bare namespace lookups such as
`ts.SyntaxKind[node.kind]` are listed without a count.

## Classification Rubric

`facade-candidate` means the file uses common compiler-API operations that can plausibly
move behind named FormSpec helpers: AST kind guards, `TypeFlags` and `SyntaxKind`
probes, type-to-string formatting, symbol lookup, path traversal, or child walking.
These are the normal migration targets for the eventual facade.

`host-boundary-exception` means the file participates directly in a TypeScript host
contract such as a tsserver plugin, a TypeScript program construction boundary, or an
ESLint parser-services bridge that hands FormSpec `ts.Program`, `ts.Node`, or
`ts.Type` values. These sites may still use helper functions internally, but the
host-facing contract itself cannot be hidden completely.

`public-api-migration` means the file exports a function, class, interface, or type whose
signature exposes `ts.*` in the generated API report or through a package-exported subpath
such as `@formspec/build/internals`. These sites need a separate public API evolution
plan, likely using the deprecation discipline in PP14, before their compiler types can be
replaced or hidden.

## Import-Style Consistency Notes

`@formspec/analysis` uses `namespace` imports in eight files and one

## Recommended First-Migration Target

Start with `@formspec/cli`. It has one direct import site, and the only value-space API
currently used there is `ts.flattenDiagnosticMessageText`; the rest of the compiler
objects are values returned by `@formspec/build/internals`. That makes it a low-risk
shake-out for a tiny named helper such as `formatCompilerDiagnosticMessage` before the
facade touches public build or analysis APIs.

Explicitly exclude the extension-callback API surface from first-migration candidates:
`ExtensionRegistry`, `StaticSchemaGenerationOptions` paths that carry extension
registries/config extensions, and analyzer or generator entry points whose behavior
depends on current `toJsonSchema(...)` callbacks should wait for #485. Migrating those
callbacks under the current raw JSON Schema shape would create facade churn that the
construct API redesign is intended to remove. A later focused `@formspec/build` slice can
target internal-only emission helpers, but only if it avoids the extension-registry and
extension-callback path.
